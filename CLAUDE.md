# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

PuntList is a mobile list app where items can be moved between lists with one tap, in addition to standard check-off behavior. Built with Flutter, backed by Firebase (Auth + Firestore).

## Running

```bash
flutter run -d chrome
```

### Local Development Prerequisites

The following files are gitignored but **required** to build and run locally. They are generated by `flutterfire configure` and contain Firebase project API keys:

- `lib/firebase_options.dart`
- `android/app/google-services.json`
- `ios/Runner/GoogleService-Info.plist`
- `macos/Runner/GoogleService-Info.plist`

If these are missing, run `flutterfire configure` with access to the Firebase project `punt-list`.

## Tests

Widget tests exercise user flows by rendering real widgets and simulating interactions. No Firebase or mocking is needed — `AppState` works without Firestore (all persistence calls are null-guarded), so tests construct screens directly with test data.

```bash
flutter test          # run all widget tests
```

### Writing new tests

- Use `createTestAppState(lists: [...])` to build state with test data (no Firestore, no init).
- Use `pumpScreen(tester, widget)` to wrap in `MaterialApp` for simple tests.
- Use `pumpStatefulScreen(tester, builder: (update) => widget)` when the test needs UI to rebuild after state changes (the `update` callback triggers `setState`).
- Use `testUpdate` as the update callback when you only need to verify state changes, not UI updates.

## App Concept

- Users create multiple named lists
- Each list item has a **checkbox** (check off) and a **move arrow →** (send to another list instantly)
- Move destinations are configured per-list in the settings page
- The move arrow is only shown on items when a destination is configured for that list
- Checked items are crossed out and sorted to the bottom

## Key Interactions

| Action | Result |
|--------|--------|
| Tap checkbox | Checks off item; crosses it out and moves to bottom |
| Tap move arrow (→) | Moves item to configured destination list |
| Tap list title | Opens rename dialog |
| Tap + on Lists screen | Creates and opens a new list ("New List") |
| Tap gear on Lists screen | Opens Settings |
| Tap item text | Inline edit |
| Uncheck a checked item | Moves back to active section |
| Checked items | Cannot be moved via → |
| Swipe item right | Indents as child of item above |
| Swipe item left | Promotes child to root, adopting later siblings |
| Press Enter while editing | Splits item at cursor into two items |
| Backspace on an empty item | Deletes it, moving the caret to the end of the previous item |
| Backspace at start of a non-empty item | Appends its text onto the previous item, deletes it, caret at the merge boundary |

There is no per-item trash icon for ordinary items — Backspace handles deletion. Parents of a sub-list keep a trash icon (see Behaviors).

## Behaviors

- Deleting a list clears it as a destination from any other list that referenced it
- New list creation immediately navigates to the new list (default name "New List")

## Sub-items

Items support one level of nesting via `parentId` on `PuntItem`. Sub-items are visually indented 48px.

### Data Model
- `PuntItem.parentId`: `null` = root item, non-null = child of that parent
- `PuntItem.withParentId(String?)`: returns a copy with a new parentId
- `DisplayItem`: rendering wrapper with `isGhostParent` flag and `isSubItem` getter
- `PuntList.activeDisplayItems` / `checkedDisplayItems`: hierarchy-aware display lists

### Indent & Promote (Swipe Gestures)
- **Indent** (swipe right, 60px): reparents a root item under the item above it, repositioning after the new parent's last child
- **Promote** (swipe left, 60px): converts a child to root; later siblings become its children; block inserted after old parent's block
- Only active (unchecked), non-ghost items can be swiped

### Checked Propagation
- **Checking a parent**: cascades to all children
- **Unchecking a child**: also unchecks parent (Google Keep "incomplete" behavior)
- **Ghost parents**: unchecked parent with checked children appears in checked section — disabled styling, non-interactive

### Reordering (Drag)
- Root items drag as a block with children
- Cannot split another parent-child group
- Child items reparent based on item above drop position

### Punt with Sub-items
- **Punting a child**: moves to destination, creating a ghost parent shell if needed (matched by parent's original ID)
- **Punting a parent**: moves parent + all children; replaces any existing ghost shell in destination

### Delete with Sub-items
- Deleting a parent cascades: removes parent and all children
- **Backspace-delete is exempt for parents-with-children**: Backspace at offset 0 does nothing on a parent of a sub-list. Because such items have no Backspace entry point, they (and ghost parents) retain the per-item trash icon as their delete affordance. Ordinary (childless) items have no trash icon.

## Backend Architecture

Firebase Auth + Firestore. All business logic runs client-side; the DB is a dumb persistence layer. `AppState` (`ChangeNotifier` + `Provider`) updates local state optimistically, then fires Firestore writes; the SDK queues and retries them, including offline. No real-time listeners in v1 — cross-device sync happens on next launch via the disk-backed cache.

See [`docs/backend.md`](docs/backend.md) for rationale (why Firebase/Firestore, alternatives considered), the data model, design-decision table, security rules, and auth flow.

- [x] Keyboard flash between items — pressing Enter on an item to create the next one makes the keyboard quickly close and reopen; keep it open. Cause: each `ItemTile` owned its own `FocusNode`, so on split the old field's node was dropped and the new field requested focus a frame later, leaving a focus gap that closed the soft keyboard. Fix: `ListViewScreen` now owns a single `_editFocusNode` shared by all active tiles (passed as `ItemTile.sharedEditFocusNode`). Only one tile renders a `TextField` at a time, so they safely reuse the node; on Enter-to-split the new tile binds the same already-focused node, so focus is never released and the keyboard stays up. The node is optional — when null (isolated `item_tile` widget tests) each tile falls back to an internal node. In `item_tile.dart` / `list_view_screen.dart`.
- [x] Remove the trash icon — Backspace on an empty item should delete it instead. if you press backspace and you're at the beginning of the text but the item still has characters, it gets appended to the previous item. Does not apply to parents of sub-lists. The per-item trash icon is removed for ordinary items. While inline-editing an active item, `ItemTile` detects Backspace-at-start with a zero-width-space **sentinel** (`ItemTile.editStartSentinel`) prepended to the edit buffer — a controller listener fires the screen's `onBackspaceAtStart` when the sentinel is deleted. (Sentinel, not `Focus.onKeyEvent`, because mobile soft keyboards deliver Backspace as a text delta, not a raw key event, so a key-event approach silently fails on Android/iOS.) `AppState.backspaceAtStart` then: (a) if the item is empty, deletes it and focuses the **end** of the previous active-display item; (b) if it still has text, appends that text onto the previous item, deletes this item, and places the caret at the **merge boundary** (the length of the previous item's original text, i.e. result is `previous.text + this.text`); (c) no-ops for the first item or for a parent-of-a-sub-list (`hasChildren`). Focus/caret restoration reuses the shared `_editFocusNode` + `_autoFocusItemId` plumbing — new `_autoFocusCursorOffset` carries the caret position, and `ItemTile` now re-applies autoFocus in `didUpdateWidget` so the **reused** previous tile re-enters edit mode (split creates a new tile, so its `initState` path was sufficient before; merge focuses an existing tile). The net-zero merge bypasses the character-limit check (text-preserving, like split). **Parent deletion:** parents-with-children are exempt from Backspace-delete and would otherwise lose their only entry point, so `ItemTile` keeps the trash icon *only* when `hasChildren` is true (this also naturally covers ghost parents). Tested in `list_view_screen_test.dart` (delete-empty, merge, parent no-op, parent trash affordance) and `item_tile_test.dart` (plain item has no trash, parent-with-children shows trash).
